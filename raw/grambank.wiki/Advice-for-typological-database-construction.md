Based on Grambank, here are some principles for the construction of typological databases that we recommend other projects consider. Below are also general project management advice that may be relevant outside of linguistics or even outside academia.

##  Advice when making a cross-linguistic database

* Consider the specific research questions you want to address *first* before designing features
* Consider how many and which observations (languages/cultures etc) you want to include to produce high-quality research. Is it  possible and necessary to include as many as possible? Which observations are most relevant to your research question? Do you want to do an even sample over families, areas and/or known disparity? Which observations are described enough to include? Consider samples such as the [Standard Cross-Cultural Sample](https://en.wikipedia.org/wiki/Standard_Cross-Cultural_Sample) for cultures, [WALS 100 or 200 samples](https://wals.info/languoid/samples/100), [Glottolog's data on available descriptions](https://glottolog.org/langdoc/status) and [available language phylogenies](https://github.com/D-PLACE/dplace-data/tree/master/phylogenies).
* Run a pilot phase of the questionnaire before taking on the full sample of observations you want to cover. In this pilot phase, take a smaller sample and go through all of the features. Once complete, consider if you need to change definitions, tighten up documentation etc. The pilot phase preferably should include an inter-coder reliability test, i.e. the same observations coded independently by different people.
* Write short, medium length and long descriptions for each feature. They should be practically oriented and continuously updated as you face coding challenges. Review our [Grambank feature wiki articles](https://github.com/grambank/grambank/wiki/List-of-all-features) as a suggestion for format. Short = the feature question itself, medium = the summary in the wiki article and long = the entire wiki article with procedure etc.
* Specify valid values for each feature clearly.
* When you can, opt for aggregating data later. For example, have a set of several binary features instead of one asking for what the "major" strategy is. Do not reduce information too early.
* In the case of grammar, consider systematically incorporating coding of form as well as pattern, i.e. when you ask "is there a past tense suffix?" also have a space to type in what the actual suffix looks like.
* Consider if you can incorporate confidence in coding or proportion of variation as an extra variable for each data point (see APiCS section on relative importance [here](https://apics-online.info/about)). Take care to ensure consistency here, different people have different default levels of confidence and different underlying datasets (corpora) can vary in genre etc such that the promotions are not comparable.
* Avoid features or feature values that capture an "other" category, e.g. "not any of the other strategies". Negatively defined categories are dangerous.
* If there are logical dependencies or other relationships between features, document this and make sure that coders and end-users are aware (e.g. [example](https://github.com/grambank/grambank/wiki/Logical-dependencies-between-features))
* Make clear how people should deal with the absence of evidence (e.g. this [article](https://github.com/grambank/grambank/wiki/Absence-of-evidence)). If possible, have different codes for when the information is not present versus when the absence of a feature is explicitly mentioned. It can be hard to tell apart missing information from inferred absence, do what you can to mitigate this.
* Assign each feature a unique ID. It does not need to be meaningful or follow in a neat sequential order with other features. It needs to be unique and stable. Never change it.
* The order of features and data points are irrelevant. No description or data point comment should contain "as above". Users should able to reorder or subset data as they please later without losing information.
* If the data is linguistic or cultural, consider releasing your data as a CLDF-dataset (possibly in addition to other formats). You don't need to curate your data in CLDF from the start. [Here is a "for dummies"-guide to CLDF](https://github.com/dlce-eva/dlce-eva/blob/main/doc/cldf_for_dummies.md). If you don't make a CLDF-release, consider making a public release of the data in an easily accessible non-proprietary format (i.e. not Microsoft Excel or SPSS-data files) in a long term stable place like [Zenodo](https://help.zenodo.org/docs/get-started/).
* Be consistent. Features should be coded the exact same way in every language regardless of family or region.
* Nominate an expert for each feature who has the final say on definitions and applications. Do not wait for group consensus to form. It is likely that there will never be group consensus. Strive instead for consistency. If one or a small set of experts are responsible for the definition of specific variables, that is a way of achieving consistency over time and observations.
* Code different observations (societies/cultures/informants/languages/dialects/doculects) each in separate tables with at least these columns: Feature_ID, Value, Comment, Source. Give the table a unique name that links to the observation, for example glottocodes in case of languages, source name in case of doculects etc.
* Encourage comments. See example of [our comment guidelines here](https://github.com/grambank/grambank/wiki/Guidelines-for-writing-comments). Good comments will help you in future when you try to understand the coding.
* For each datapoint, specify the source used (including page). 
* Keep track of who did what by assigning everyone a unique abbreviation and use this in the data dabble in a separate column to indicate who did what. For example, in the table below "AR" and "HS" refers to difference coders: 

Feature_ID|Glottocode	|Value|	Source|	Comment	|Contributed_Datapoints|
--|--|--|--|--|--|
GB025	|biwa1243|3	|McElvenny (2006: 10-12)|		|AR
GB027	|biwa1243|1|	McElvenny (2006: 34)|	There is no evidence for a conjunction construction, the comitative is used to express the co-ordination of two noun phrases.	|AR;HS

* Have a bibTeX-file with all references cited. You can use [hh.bib](https://github.com/glottolog/glottolog/blob/master/references/bibtex/hh.bib) (a part of Glottolog) as a base and then add an bibTeX-file for sources not found in hh.bib.
* Match sources in the coding sheet to references in the bibliography via culture/language identifiers, author name and year. This can help avoiding unfortunate mishaps such as someone assigning a file the wrong glottocode because then the linking to references will break. For more info on how this is done in Grambank, go [here](https://github.com/grambank/grambank/wiki/Referencing-sources-in-Grambank).
* Avoid using personal correspondence-referencing when possible, it is better if people outside of your project can verify the information independently.
* Do basic quality checks at every submission of new coding  ([see our automatic checks here](https://github.com/grambank/grambank/wiki/Automatic-quality-checks)), answering questions such as:
   - can every source used in coding be mapped to a reference in a bibliography? [more guidelines on referencing here](https://github.com/grambank/grambank/wiki/Referencing-sources-in-Grambank)
   - all are values valid? (i.e. no "XY?")
   - does every value have a source?
   - are there any combinations of features that are impossible? (e.g. "languages which have a dual-augmented pronoun system can't also not have dual number in pronouns")
   - are there any values that are very rare? If so, flag it at submission to double-check that it's real and not a mishap (e.g. "trial number is rare, are we sure about this?")
* Outline a clear workflow for who does what in terms of coding, quality control, project management and curation - including potentially setting an end-point for when the dataset isn’t curated anymore. The World Atlas of Language Structures was not curated past 2013 for example.
* Ensure that people can work in peace and get issues resolved by having standing meetings for everyone involved in data collection at least once a month where people can discuss difficulties in coding
   - have one document with an agenda that everyone can edit where people add issues they have before the meeting
   - if it takes longer than 20 minutes of solitary thinking to make a coding decision, then the coder should note it down in the group agenda, leave it to a group meeting and move to the next task in the meantime - otherwise it is possible to get stuck
   - don't discuss problems immediately when they occur (not in person nor over email), make dedicated time for it in everyone's schedule by using the regular meetings
   - in the description of the problem, be very explicit and clear what exactly the problem is and what information you already have ("I can't decide what to do about tense affixes. Here's what the author says, they look like mood to me")
   - keep meeting notes after as well as an agenda, write down what the solution was to each issue - however brief
   - make the person keeping the notes and the person chairing the meeting two different people
   - for meetings, focus on problem-solving and stick to concrete issues (not hypotheticals)
   - see more on our coding problem-solving workflow [here](https://github.com/grambank/grambank/wiki/Encountering-coding-problems-and-solving-them)

## General project management

The Grambank project has been ongoing for many years and involved a large number of people, many of whom are not in the same physical location. The work is complex and constantly generates challenges. This has necessitated the development of a remote network and project management practices to ensure quality work and happy workers. The project management style has some similarities with [the scrum approach in software development](https://en.wikipedia.org/wiki/Scrum_(software_development)), [Extreme programming (XP)](https://en.wikipedia.org/wiki/Extreme_programming) and meeting techniques from Swedish non-profit leisure associations (e.g. Sverok and Unga Forskare). Here are some more general points that may be transferrable to other projects facing similar challenges, be they in academia, industry or elsewhere.

* **There is no such thing as a flat hierarchy**. It is better to explicit and clear about who has the power to make what decisions than to hide behind the false pretense that power is distributed equally. This makes it easier to move forward, and it also facilitates locating where the responsibility _and_ accountability lies. Leaders can and _should_ take input, but in the end, everyone should know where authority lies for specific areas and tasks. It may be awkward and embarrassing to be explicit about power relations, but it is 100% worth it. Draw a tree diagram if you need to. This is also true of academic papers, there needs to be at least one person who is mainly in charge and everyone should be 100% clear on who that is.
* **Reject suggestions without being dismissive**. Not all ideas are good or feasible. Everyone can make suggestions that are bad or otherwise not suitable. The group, and in particular the person in charge, needs to be able to say "no" without also embarrassing the person who made the suggestion. Acknowledging the good parts and outlining why the idea is not _feasible_ as opposed to inherently _bad_ is often the best way. In addition, it is preferable to offer alternatives compared to only shooting down ideas. Group members who feel dismissed will be less happy, produce less good work and give less input to discussions etc. This is bad for the group's goals.
* **Move forward also on tough decisions**. Because it is safer to weigh in on less important things, groups can sometimes find themselves spending a disproportionate amount of time on trivial matters (Oxford commas, color schemes etc) and avoid more heavy agenda items. This is also known as the ["Law of Triviality" or bike-shed-effect](https://en.wikipedia.org/wiki/Law_of_triviality). It is necessary for the leader in the group to identify when this is happening and find a way to move past it. For example, trivial matters could be relegated to pre-meeting text-based communication only whereas more important topics could be discussed in precious face-to-face meetings.
* **Encourage discussion, within reason**. It is good if project members feel comfortable to suggest ideas, give criticism etc. However, avoid discussions related to solving hypothetical problems that have not occurred yet (see also ["You Ain't Gonna Need It (YAGNI)" in XP](https://en.wikipedia.org/wiki/You_aren%27t_gonna_need_it)). Group discussions should have an end-point that results in a decision on a specific practical matter. The leader should make clear when that point is reached and exactly what the decision is (e.g. "Regarding model fit selection, we are going with WAIC. Next item.").
* **Scale your emotions - it's probably not life-and-death**. It is important to be professional and take things seriously, but don't go overboard. If someone makes a mistake, something goes wrong or something is urgent - take action but make sure you're not stressing everyone out unnecessarily. Your project is probably not determining life-and-death scenarios, so scale the emotional response accordingly. Focus on the next practical steps of solving the problem, who is doing what and who is in charge and how you can improve project documentation to avoid the problem in future. If there is too much negative attention when a mistake happens, people may not want to acknowledge mistakes which in turn is bad for the project.
* **Be encouraging**. It's easy to only focus on negative criticism when working together because you want to "fix it" and move on. You may take for granted that people know that you think they're fundamentally doing a good job. This may not be the case. Don't neglect giving positive feedback to your colleagues (especially those under you in a hierarchy) and reassuring them that while you end up discussing a lot of difficult challenges and perhaps disagreeing on key issues  - they are in fact doing alright. If this is lacking, you can create a tense atmosphere. Failure to make people feel secure in the group and their roles can lead to [imposter syndrome](https://en.wikipedia.org/wiki/Impostor_syndrome), which is destructive for the individual and the group.
   * **Advanced level**: try to end meetings with saying something nice to your collaborators (e.g. "I really like working with you", "that was a very good idea"). Don't fake it though, only do it if it's earnest.
   * **Advanced level**: if someone is contributing something valuable, consider telling other people - especially superiors (e.g. "Maddie has been very helpful lately regarding X", "Catherine is such a good addition to our team because of her skills in Y").
* **Assume good faith** Start from the assumption that people are acting in good faith and that they are approachable in regards to feedback and suggestions. Misunderstandings can generate a feeling that people are intentionally uncooperative. This is very, very rarely the case. Assume instead that people are trying their best within the parameters of the situation, their skillset and understanding. If friction occurs, take a pause and reflect on the variables involved. Deconstruct the situation together by spelling out assumptions and conditions and work forward together.
* **Aggregate decisions into dedicated time slots**. It is generally not acceptable to interrupt someone else's work right away when you have an idea or a problem (unless you have already mutually established that this is okay, of course). Keep a running agenda/list that everyone can add to as they encounter challenges/have ideas is often good. Dedicate a specific time to go through these items together. In that way, all group members can move forward safe in the knowledge that their issue will be addressed while at the same time, everyone's time and mental energy is respected and people can adequately prepare for the meeting. 
   * **Advanced level**: label all agenda items as either _information dissemination_, _discussion_ or _decision_. These are generally the _only_ three kinds of items you should have in a meeting. Preferably all _discussion_-items should swiftly be made into _decisions_. If possible, outline the possible options for the decision already as you are setting the agenda (e.g "We need to make a decision about X. Our options are Z, Y and other"). If a decision is made, write explicitly who is responsible for taking action and reporting back at the next group meeting (a type of _information dissemination_ agenda item).
* **Plan in levels**. Some plans are for things that need to be done very soon, others are less urgent. Make clear what the expected time period is for each goal in your project. Just because something isn't urgent it does not mean it doesn't deserve a plan, even if it's just a preliminary sketch. Making long-term _and_ mid-term plans allows everyone in the project to plan their work life (other research, teaching etc) and non-work life (family care, holidays etc) accordingly. Focussing almost exclusively on urgent matters is often very stressful and will most likely lead to reduced mental health and poor work quality. It is not plausible that _all_ tasks within your project are either to be done immediately or at some undefined point in the future. Consider classifying tasks explicitly into the three categories below. If all of your items are in either the ASAP and Later-categories, consider carefully if some can be adjusted to the Mid-distance slot. Keep the ball rolling, but don't stress everyone out.
   * As Soon As Possible (ASAP)
   * Mid-distance (within the next 4-12 weeks)
   * Later (within a year) 
* **Keep track**. When decisions are made, keep a written record of it. If an issue needs a follow-up, keep it on the agenda for the next meeting. Using online collaborative documents for agendas makes this easier. Do not rely on human memory, it is fallible and insufficient. You'll remember things differently, forget details etc. 
* **Follow up**. Keep a list of all ongoing goals and regularly go through them, even if that means the person in charge of the goal only says "nothing has happened since last time we checked in" or "we have decided to drop this". Again, do not rely on human memory only. Following up is necessary not because of lack of trust in people doing their jobs, regardless of trust it is necessary to re-estimate the task continuously and for the leader to have a good overview of the project.
* **Acknowledge failure and build trust and mutual respect**. We learn from failures more than from success. All group members should be made to feel comfortable sharing mistakes and problems. If you are part of the project, that already means that you are good enough and have the group's trust. Your leader should inspire this trust that you are good enough. Failure to do this can lead to [imposter syndrome](https://en.wikipedia.org/wiki/Impostor_syndrome), which is destructive for the individual and the group.
* **Terminate explicitly rather than let slip through the cracks**. If something isn't working out or you decide to not do something, be explicit about it. Don't let projects or issues fall into the limbo state of "Are we still doing that or did we decide to drop it?". Not everything was meant to be, it is okay to say goodbye and let go. Do not assume that people will understand implicitly that something should be dropped. Make it clear to _everyone_ that something is no longer happening/not moving forward.
* **Be kind and be flexible**. Be nice to people and be willing to compromise - not everything will go exactly your way. You do not need to die on every hill, let people do it their own way sometimes. Reign in your need for control.
* **Telepathy is not real (yet)**. Your colleagues are not mind-readers, you will need to be explicit about many things that you may believe are obvious. Conversely, people may tell you things you think are obvious and you may find that patronising. It is possible (but not necessary) that it is not meant to be patronising, perhaps they are just being extra careful about being explicit to avoid misunderstandings. State clearly what you know, what you assume and the conclusions you draw. See also [the rubber duck method for solitary problem solving](https://en.wikipedia.org/wiki/Rubber_duck_debugging) (explaining from very basic assumptions, spelling out expectations and comparing to actuality).
* **Social norms are not universal**. Politeness, directness, how to make requests etc are not universal. Be mindful of the cultural diversity in your group and strive to minimise unnecessary friction due to people having different "baselines" for politeness, annoyance expression etc. Some people prefer more indirect communication, and for them it is possible that direct requests are understood as rude and aggressive. Each group member should take mental stock of the variation in the group, aim to adapt communication such that it reduces friction and strive towards patience. Do not assume malice when it might just be a misunderstanding.
* **E-mails are tricky**. Everyone has their own way of dealing with e-mails and e-mails will probably feature a lot in your project management. At the very start of your project, have a frank and explicit conversation about how e-mails will feature in the group work. Possible guidelines are:
   * Use informative headings with distinct keywords to help people find information in their inboxes.
   * Don't expect replies within unreasonable time-frames (consider people's time zones, care responsibilities, vacation, other workload, week-ends, sleep etc.). Reasonable short time-frames are probably 2-6 days.
   * Be mindful of who receives the message when you reply-all.
   * Avoid long threads, when possible use short in-line responses and use meetings for anything that is generating paragraph-length e-mail threads.
   * Not all institutional email addresses are reliable long-term, some institutions remove accounts and do not allow forwarding once the person is no longer employed there. This is for example the case for Max Planck-emails. Therefore, you may need access to multiple email addresses for the same person even if one is the official and preferred one at the time.
   * Avoid using email for document coordination, especially _do not email files back and forth_. Find a shared collaborative solution (Google Docs, GitHub, Dropbox, Nextcloud etc) and set up a folder structure there with the files you are working on together. 
* **Centralise work and information**. Reduce the number of platforms (Google, GitHub, Tableau, Slack etc) you use and the number of structures within them (folders, repos, tables etc). The fewer places people have to look for information, the less cognitively taxing. If you have many, keep one document that lists them (e.g. [our GitHub repos wiki article](https://github.com/grambank/grambank/wiki/Git-repos-structure)). When possible, explore if you can aid people by using fewer tools. For example, is it possible to make the group calendar viewable in Google Calendar, Apple's iCal etc instead of forcing people to use one of them? If so, group members are less likely to miss events as they do not need to keep track of multiple calendars. [Keep It Simple Silly](https://en.wikipedia.org/wiki/KISS_principle).
* **Don't Repeat Yourself (DRY)**. Don't store the same information in several places, they can get out of sync which can cause misunderstandings and confusion. Keep information in one place and if necessary use links to point there from other places. The most up-to-date description of your project should exist in _one_ place only. This is [a principle in software development](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself), but is relevant outside of programming too.
